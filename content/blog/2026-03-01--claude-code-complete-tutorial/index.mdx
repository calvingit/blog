很多人把 Claude Code 当成“更聪明的自动补全”，结果是：能快一点，但不稳定，也容易返工。

这篇总结最重要的启发是：决定产出的不是模型本身，而是你如何规划任务、约束上下文、以及建立验证闭环。换句话说，真正要升级的是工程方法，不只是工具。

## 一、先思考，再输入

作者反复强调一个点：别一上来就把需求一股脑丢给模型。

在真正动手前，先明确三件事：

1. 目标结果是什么
2. 边界条件是什么
3. 成功标准是什么

在 Claude Code 里先进入 plan mode（`Shift + Tab` 两次），用 5 分钟梳理方案，通常能省掉后面几小时调试。

### 为什么这一步最关键

因为 AI 对“模糊任务”会自动补全太多你没要求的东西。比如：

- 模糊输入：`帮我做一个认证系统`
- 清晰输入：`基于现有 User 模型做邮箱密码登录，Session 存 Redis，24 小时过期，并保护 /api/protected 路由`

后者不仅更可控，返工成本也会低很多。

## 二、把 CLAUDE.md 当作项目的“长期记忆”

作者把 `CLAUDE.md` 定位为每次会话都会读取的入场说明。

高质量的 `CLAUDE.md` 不是新人文档，而是“项目里真正会影响实现质量的信息”：

- 关键命令与流程
- 约束原因（不仅是规则）
- 常见错误与禁区
- 团队做过的关键决策

### 实战原则

1. 保持短而精
2. 只写项目特有信息
3. 解释“为什么”
4. 持续更新（同类纠错出现第二次就写进去）

这样它会逐步演变成可复用的工程资产，而不只是说明文件。

## 三、上下文窗口不是越满越好

作者给了一个非常实用的经验：上下文质量衰减往往在“远没满”时就开始。

因此要主动管理上下文，而不是等模型“自己扛住”：

- 一个会话只做一个任务
- 复杂任务写入外部记忆文件（如 `SCRATCHPAD.md`、`plan.md`）
- 上下文发散时及时 `/clear`
- 必要时先 `/compact` 再精简关键信息重启

核心心智模型：模型是“近似无状态”的，你不给，它就不知道。

## 四、提示词能力决定产出上限

作者观点很直接：坏结果多数来自坏输入，而不是模型不够强。

更有效的提示词结构应包含：

1. 明确任务目标
2. 明确不要做什么（防止过度设计）
3. 明确性能或业务约束
4. 明确完成定义和验收方式

例如你可以直接加上：

```txt
保持实现简单，不要新增我没要求的抽象。
优先单文件改动，必要时再拆分。
```

这类反约束对抑制“过度工程化”非常有效。

## 五、按任务切换模型，而不是盲目统一

文中给出的实用策略：

- Opus：用于规划、权衡、复杂推理
- Sonnet：用于按既定方案高效执行

如果你先用 Opus 做架构决策，再用 Sonnet 执行细化，通常能在质量和成本之间取得更好的平衡。

## 六、不要只用交互式，尽量系统化

作者建议把 Claude Code 纳入自动化流程，而不是仅在终端里“一问一答”：

- 用 headless 模式串联脚本
- 用 Hooks 自动跑格式化、类型检查、测试
- 用 MCP 对接外部系统（GitHub、Slack、数据库等）
- 把重复任务沉淀为 slash commands

当这些机制建立起来后，AI 不是一次性提速，而是持续复利。

## 七、Claude 卡住时，优先“换策略”

常见低效模式是：同一个解释重复三遍，模型还是走偏。

更有效的做法：

1. 清空会话重来
2. 把任务继续拆小
3. 用最小示例“示范成功样子”
4. 重述问题框架（换一种建模方式）

重点不是“更大声地解释”，而是“改变问题表达方式”。

## 八、可直接复用的执行清单

如果你今天就要落地，可以按这个顺序：

1. 为当前需求先写 plan，再开始编码
2. 新建或精简 `CLAUDE.md`（只留高价值规则）
3. 给提示词补齐目标、约束、反约束、验收
4. 一项任务一个会话，避免上下文污染
5. 接入最基础的自动验证命令（lint/test/build）
6. 把高频工作转成可复用脚本或命令

## 总结

这条“完整教程”真正传达的不是某个技巧，而是一种方法论：

- 先想清楚，再让模型执行
- 用项目记忆和约束提高一致性
- 通过上下文管理保持稳定输出
- 把 AI 融入工程系统，而不是当临时助手

当你把这些动作变成习惯，Claude Code 才会从“偶尔惊艳”变成“稳定产能”。
