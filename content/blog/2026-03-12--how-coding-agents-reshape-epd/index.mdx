软件团队里常说 EPD，也就是 Engineering、Product、Design。三个角色分工不同，但目标其实很一致：做出真正能解决问题的软件。

这篇文章里我最认同的，不是“人人都能写代码”这种表层判断，而是另一件事：当编程 Agent 把实现成本压低后，团队里最稀缺的能力就变了。以前稀缺的是把东西做出来的人，现在更稀缺的是能判断做什么、怎么评审、哪里会失控的人。

换句话说，瓶颈正在从实现，转向评审。

## PRD 没死，只是不再是流程起点

原文里有句很抓眼球的话：`PRD 已死`。这句话很容易被读偏。它的意思不是以后不需要需求表达，而是传统那套 `PRD -> 设计稿 -> 代码` 的线性接力，没以前那么重要了。

原因不复杂。以前把一个想法做成原型很贵，所以 PRD 是起点，大家要靠文档把意图一级级往下传。现在不一样了，很多想法可以先直接做成能跑的东西，再拿着原型进入讨论。文档没有消失，只是从“实现起点”变成了“评审配套”。

所以我更愿意把今天的 PRD 理解成一种评审材料。它可以是一段结构化说明，也可以是一组约束，甚至就是围绕原型写的几段决策记录。形式会变，但“把为什么说清楚”这件事没变。

:::tip[核心判断]
如果实现已经足够便宜，那么真正稀缺的就不是“有没有人能做出来”，而是“谁能判断这东西值不值得做、架构是否合理、体验是否可用”。
:::

## 瓶颈为什么会从实现转向评审

实现一旦便宜，原型就会大量出现。问题也跟着来了：

1. 能跑，不等于架构合理
2. 能用，不等于解决了正确的问题
3. 能上线，不等于体验足够好

过去原型少，是因为做一个原型本身就贵。现在原型多了，真正贵的是评审时间。工程要看技术债和扩展性，产品要看问题有没有抓准，设计要看交互是不是顺手。最后大家都在抢同一件事：判断这个东西该不该继续往前推。

## 通才会更值钱，但不是因为“什么都懂一点”

这里还有个容易被过度解读的点：通才会更值钱，但不是那种“什么都懂一点”的通才。真正占便宜的，通常是在一个方向有深度，同时能和另外两个方向顺畅对话的人。

原因很现实。以前这类人就很有价值，只是大量时间花在实现上。现在实现阻力下降后，他们可以更快把判断变成原型，再自己迭代一轮。单人杠杆自然就变大了。

## 编程 Agent 会变成必选项

这一点我基本同意。以后不会写代码的人未必马上出局，但不会用编程 Agent 的人，大概率会先吃亏。

产品可以先做原型再谈需求，设计可以直接在代码里试体验，工程也能把重复实现交出去，把时间留给系统设计和评审。它不要求每个人都变成传统意义上的工程师，但要求每个角色都能把 Agent 当执行接口，而不是继续只会往下游派单。

## 真正被放大的，是判断力

实现更便宜，放大的不只是好想法，还有坏想法。

这也是原文里那句“好的 PM 会更好，差的 PM 会更差”的背景。错的方向现在也能很快做出像样的原型，甚至更危险，因为东西一旦做出来，团队就更容易被沉没成本拖着走。

所以最后比拼的还是判断力。工程判断系统边界和技术债，产品判断问题值不值得解，设计判断体验是不是顺。Agent 只会放大这些判断，不会替你做掉。

## 专业化不会消失，但门槛会更高

我比较认同这篇文章的地方，是它没有直接滑向“以后只剩通才”这种轻飘的结论。

专业角色不会消失，反而会更贵。只是门槛会往上抬，因为你不只要专业够深，还得能在更快的节奏里持续做判断：

1. 评审速度更快
2. 沟通能力更强
3. 对其他角色的约束和目标有基本理解
4. 能在 Agent 加速后的节奏里持续做判断

## 组织里可能会更明显地分成两类人

原文最后给出了一个很有启发的划分：建设者和评审者。

建设者负责把想法尽快变成原型，让讨论有抓手。评审者负责筛选、判断、定方向，避免团队在看起来不错的东西上走太远。

我不把它理解成正式的组织结构，更像是能力重心的变化：

1. 有些人会越来越偏向“快速构建”
2. 有些人会越来越偏向“高质量评审”
3. 大多数人还会落在中间，但会被迫往某一侧更清楚地站队

这也解释了为什么工程、产品、设计的边界会继续变松。不是角色名没了，而是每个人都得补上原来不属于自己的那一段能力。

## 我自己的几个结论

如果把这篇文章压缩成对个人最有用的建议，我会留下面这几条。

### 1. 先别把 Agent 当成“更快写代码的工具”

它当然能提速写代码，但更大的变化是，它让原型变得很便宜，连带着把团队讨论、协作和评审的方式都改了。

### 2. 学会评审，会比学会生成更稀缺

会生成的人会越来越多，会判断的人不会自动变多。后者大概率更值钱。

### 3. 产品意识会变成所有角色的基础能力

不理解用户问题、功能边界和优先级，过去还能被流程遮一遮，之后会被 Agent 直接放大。

### 4. 不管你现在在哪个角色，都应该补另一侧能力

工程师要补产品和设计判断，产品和设计也该补实现和系统约束。不是为了角色混同，而是为了减少来回损耗。

## 结语

“编程 Agent 会不会替代谁”当然是个热闹话题，但我觉得没那么重要。更重要的问题是：实现不再昂贵之后，团队里到底什么能力会重新变得稀缺？

这篇文章给出的答案其实很明确：

1. 需求表达不会消失，但会从流程起点变成评审搭档
2. 实现门槛下降后，评审会成为新的瓶颈
3. 系统思维、产品意识和跨角色判断会变得更重要
4. 角色边界会变松，但高水平专业能力不会失去价值

如果今天还把编程 Agent 只看成工程师的提效工具，我觉得是低估了它。它真正改变的，可能不是谁来写代码，而是谁来定义问题、谁来判断质量、谁来推动协作。
