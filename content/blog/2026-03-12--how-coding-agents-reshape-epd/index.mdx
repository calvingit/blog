软件团队里常说 EPD，也就是 Engineering、Product、Design。三个角色分工不同，但目标其实很一致：做出真正能解决问题的软件。

这篇文章最值得记住的判断，不是“人人都会写代码了”，而是另一层更深的变化：当编程 Agent 把实现成本压得很低之后，团队最稀缺的能力不再只是“把功能做出来”，而是“判断做什么、评审做得对不对、确保系统不会失控”。

换句话说，瓶颈正在从实现，转向评审。

## 为什么这个变化值得认真看

过去一套相对标准的软件流程，大致是这样：

1. 产品提出想法
2. 产品经理写 PRD
3. 设计师做设计稿
4. 工程师把设计稿实现成代码

这套流程背后的前提很朴素：做设计和写代码都很贵，所以必须分工，必须交接，必须用文档把意图传下去。

编程 Agent 改变的正是这个前提。它让“把想法变成一个能跑的原型”突然变得很便宜。原来最花时间的那一段，不再是最大的障碍。于是团队里很多习惯性的分工和交接方式，也会跟着变化。

这并不意味着产品、设计、工程会消失。相反，它们会更像三种评审能力，而不只是三段接力流程。

:::tip[核心判断]
如果实现已经足够便宜，那么真正稀缺的就不是“有没有人能做出来”，而是“谁能判断这东西值不值得做、架构是否合理、体验是否可用”。
:::

## PRD 已死，但需求表达没有死

原文里最醒目的一句是“PRD 已死”。这句话如果只看表面，很容易误解成“以后不需要需求文档了”。更准确的理解是：

传统那种 `PRD -> 设计稿 -> 代码` 的线性流程，正在失去中心地位。

因为现在很多时候，最先出现的可能不再是一份 PRD，而是一个原型。有人有了想法，直接借助编程 Agent 做出可运行的东西，然后拿着原型进入讨论和评审。

但这不代表需求表达不重要。恰恰相反，需求表达仍然非常重要，只是它的角色变了。

以前文档更像是“实现的起点”。现在文档更像是“评审的搭档”。原型告诉别人你做了什么，文档解释你为什么这么做、哪些地方是有意为之、哪些是暂时妥协、接下来准备怎么推进。

这也是为什么作者会说“PRD 已死”之后，又接一句“PRD 万岁”。死掉的是旧流程，不是需求表达本身。

我觉得这里最有意思的一点是，未来的需求文档未必还是传统 PRD。它完全可能变成一份结构化的 prompt、一段带版本记录的说明，或者一组围绕原型展开的设计约束。

## 瓶颈为什么会从实现转向评审

实现变便宜之后，原型会大量出现。问题也随之而来：

1. 能跑，不等于架构合理
2. 能用，不等于解决了正确的问题
3. 能上线，不等于体验足够好

过去做一个原型要花很多时间，所以团队能看到的候选方案其实有限。现在谁都能更快地做出东西，进入评审队列的原型数量会明显增加。对于工程、产品、设计三类角色来说，最贵的时间反而开始花在“看这些东西值不值得继续推进”上。

从工程角度，要看架构是否稳、是否可扩展、是否会留下高昂技术债。

从产品角度，要看这个功能是否真的切中用户问题，而不是只是看上去像个功能。

从设计角度，要看交互是否自然、界面是否直观、是否真的好用。

这意味着 EPD 不再只是“不同角色各做各的”。它们会一起围绕原型做判断。谁能更快、更准地评审，谁的价值就更高。

## 通才会更值钱，但不是因为“什么都懂一点”

原文里另一个判断是：通才会比以前更有价值。

这个判断成立，不是因为通才突然比专家厉害了，而是因为跨角色沟通的成本正在被重新计算。

一个同时具备产品感觉、设计判断和工程执行力的人，过去也很宝贵，只是他仍然要花很多时间在真正的实现上。现在实现本身的阻力下降了，这类人可以直接把更多判断转成原型，再快速自我迭代。于是单人影响力会被显著放大。

但这里有个容易被浪漫化的误区：通才更值钱，不等于“粗糙的全能感”更值钱。真正有价值的通才，往往不是每个方向都懂个皮毛，而是至少在一个方向有扎实深度，同时对另外两个方向有足够成熟的判断力。

所以这不是“大家都不需要专业能力了”，而是“只有会合作的专业能力，才更值钱”。

## 编程 Agent 会变成必选项

原文把这一点讲得很直白：使用编程 Agent 会成为必选项。

这几乎可以视为一个职业层面的现实判断。

产品经理不再只能写需求再等工程排期，而是可以先做原型验证想法。设计师不再只能停留在 Figma，而是可以直接在代码里迭代体验。工程师则可以把更多时间从重复实现里抽出来，投入到系统设计和评审上。

这并不意味着每个人都必须成为传统意义上的工程师，而是说：每个角色都要学会把编程 Agent 变成自己的执行接口。

如果别人在用 Agent 快速验证、快速沟通、快速推进，而你还停留在只会把问题往下游扔，那角色价值一定会被重估。

## 真正被放大的，是系统思维和产品意识

实现更便宜以后，最容易被放大的，不只是好想法，还有坏想法。

原文里有一句很扎实：好的 PM 会更好，差的 PM 会更差。因为错误的想法也能更快地做出原型，而且一旦东西已经“做出来了”，团队就更容易因为沉没成本和上线惯性把它继续推进。

所以真正重要的能力会变成两类。

第一类是系统思维。

工程需要清楚服务怎么拆、数据怎么流、系统在哪些地方会出问题。产品需要清楚用户到底缺什么，而不是只听用户嘴上说什么。设计需要清楚为什么一个交互顺手，为什么另一个交互会让人困惑。

第二类是产品意识。

编程 Agent 只能执行被表达出来的方向。如果给它的方向本身就错了，它只会更快地帮你制造更多待评审的垃圾。知道该做什么、为什么做、做到什么程度才算够，这些判断不会因为 Agent 出现而变得不重要，反而会更重要。

## 专业化不会消失，但门槛会更高

这篇文章没有走向“以后只剩通才”这种简单结论，我觉得这是它比较清醒的地方。

深度专业角色仍然会存在，而且仍然重要。顶级的系统架构师、真正理解用户问题的产品负责人、能设计完整用户旅程的设计师，都会继续有价值。

但门槛会抬高。

因为这类角色以后不仅要在自己的专业上足够强，还要：

1. 评审速度更快
2. 沟通能力更强
3. 对其他角色的约束和目标有基本理解
4. 能在 Agent 加速后的节奏里持续做判断

以前专业能力强，已经足够稀缺。以后你还得能在更快的节奏里，把这种能力稳定地转化成决策和评审。

## 组织里可能会更明显地分成两类人

原文最后给出了一个很有启发的划分：建设者和评审者。

建设者负责把想法迅速变成原型，推动小功能落地，或者把大功能先做成可讨论的版本。他们需要不错的产品意识、基本的设计直觉，以及把编程 Agent 用顺手的能力。

评审者则负责对复杂功能做深度判断。他们需要在自己的领域里具备顶级系统思维，并且能在大量原型涌入时快速筛选、快速决策。

我觉得这个划分并不是绝对的组织结构，更像一种能力重心：

1. 有些人会越来越偏向“快速构建”
2. 有些人会越来越偏向“高质量评审”
3. 大多数人则会落在两者之间，但会被迫往某一侧更清晰地靠拢

这也解释了为什么工程、产品、设计的边界会越来越模糊。不是角色名称消失了，而是大家都要补足原来不属于自己的那部分能力。

## 我从这篇文章里得到的几个现实结论

如果把这篇文章压缩成对个人最有用的行动建议，我觉得大概是下面几条。

### 1. 先别把 Agent 当成“更快写代码的工具”

它更像一个把想法快速外化成原型的系统。它改变的不只是工程效率，而是整个团队讨论问题、传达意图和做评审的方式。

### 2. 学会评审，会比学会生成更稀缺

生成能力会越来越普及，但判断能力不会自动普及。未来更值钱的人，很可能不是最会“让 Agent 干活”的人，而是最会判断“Agent 产出的东西是否值得继续推进”的人。

### 3. 产品意识会变成所有角色的基础能力

不理解用户问题，不理解功能边界，不理解为什么做，这些短板以前可能被流程掩盖，未来会被 Agent 放大。

### 4. 不管你现在在哪个角色，都应该补另一侧能力

工程师要补产品和设计判断，产品和设计要补对实现和系统约束的理解。不是为了角色混同，而是为了减少沟通损耗，提高原型和评审的质量。

## 结语

“编程 Agent 会不会替代某个角色”其实不是这篇文章最重要的问题。

更重要的问题是：当实现不再昂贵之后，团队里什么能力会重新变得稀缺？

这篇文章给出的答案很清楚：

1. 需求表达不会消失，但会从流程起点变成评审搭档
2. 实现门槛下降后，评审会成为新的瓶颈
3. 系统思维、产品意识和跨角色判断会变得更重要
4. 角色边界会变松，但高水平专业能力不会失去价值

如果你今天还把编程 Agent 只看成工程师的工具，那很可能低估了它。它真正改变的，也许不是“谁来写代码”，而是“谁来定义问题、谁来判断质量、谁来推动协作”。
