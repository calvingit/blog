这篇文章最打动我的，不是它又给 AI 编程起了一个新名字，而是它把一个老问题讲得很透：为什么很多代码库越做越累，不是因为功能变多了，而是因为每一次交付都只留下了功能，没有留下系统性的增益。

复利工程想解决的就是这个问题。它要求每一单位工程工作都让后续工作更容易，而不是更难。这个目标听起来很抽象，但文章给的落点其实很具体：你做完一件事后，除了交付功能，还得把这次学到的模式、规则、约束和检查方法回写进系统。

## 分水岭不在前三步

文章把主循环写成 `Plan -> Work -> Review -> Compound -> Repeat`。前面三步大家都熟：先规划，再实现，再评审。真正把它和“传统工程 + AI 辅助”区分开的，是第四步 `Compound`。

因为前三步只会产出一项功能，第四步才会产出一个更会做事的系统。

Every 的原文里把这一点写得很直接：如果你跳过 Compound，这仍然只是传统工程，只不过实现阶段换成了 AI。只有把本次解决方案沉淀成可复用资产，下一轮才会更快、更稳，也更少依赖个人记忆。

所以我更愿意把这套方法理解成一种交付纪律：

- `Plan` 决定这次做什么
- `Work` 把它做出来
- `Review` 把问题拦住
- `Compound` 把经验留下来

少了最后一步，前面三步的收益不会积累。

## 真正该沉淀的是什么

原文在 Compound 这一步列的动作很有代表性：记录什么有效、什么无效，把解决方案做成可检索内容，更新 `CLAUDE.md`，必要时新增 Agent 或命令，并确认系统下次能不能自动避开同类问题。

这说明作者关心的不是“写一篇复盘”，而是让系统真的学会。

我会把它压成三类资产：

- 规则资产：比如 `CLAUDE.md`、`AGENTS.md`、审查清单、风格约束
- 流程资产：比如工作流命令、专项 Agent、自动化检查
- 经验资产：比如设计文档、问题解决记录、可复用方案

这三类东西的共同点是，它们都不是为这一次交付服务，而是为了下次别再从零开始。

## 它真正挑战的是工程师的旧默认

这篇文章里有一大段“需要放旧的信念”，我觉得并不是为了制造话题，而是在纠正一个现实：很多团队明明已经在用 AI 写代码，但脑子里还是手工时代的分工模型。

比如：

- 代码必须由人亲手写出来
- 每一行都要人工盯着看才算安全
- 解决方案必须先从工程师脑子里长出来
- 代码是最核心的产物

这些说法在过去有其合理性，但在 Agent 已经能做研究、实现、测试和自检的前提下，它们会把团队重新拖回“人肉补偿系统缺陷”的状态。你不信任输出，就继续靠人工逐行兜底；你不沉淀经验，就只能反复重复同一类判断。

复利工程的意思不是“人不重要了”，而是人的价值从“亲自打字”更多转向“定义约束、做判断、建立系统”。

## 最值得采纳的四个动作

如果只保留最有工程价值的部分，我会先抓下面四件事。

第一，把团队的品味提取到系统里。命名偏好、错误处理方式、测试习惯、评审标准，不要只留在资深工程师脑子里，应该写进能被 Agent 读取和执行的地方。

第二，给系统改进留出稳定预算。原文提出过一个很激进的视角：工程时间不该几乎全投在功能上，系统建设本身也应当是主要工作。你不必照搬比例，但这个方向是对的。

第三，用护栏替代盯梢。测试、自动审查、日志、监控、可回滚机制，这些不是附属物，而是“你敢不敢放手”的前提。

第四，尽量让环境对 Agent 友好。它能不能跑测试、看日志、做截图调试、处理 Git 操作，直接决定了你还有多少事必须自己补做。

## 它适合哪些团队

复利工程最适合的，不是所有场景，而是那些会重复交付、重复踩坑、并且有长期维护成本的系统。

比如：

- 持续演进的产品代码库
- 审查和测试规则相对清晰的团队
- 问题模式容易被文档化和自动化的领域

如果只是一次性脚本、极短生命周期项目，或者问题空间还处在高不确定探索期，这套方法的收益会小很多。因为复利成立的前提，是今天沉淀的东西明天还会继续用。

## 我对这篇文章的最终判断

这篇文章表面在讲 AI-native engineering，真正重要的不是“AI 替你写代码”，而是它重新定义了什么才算工程产出。

过去我们更容易把功能当作产物。现在如果还只交付功能，不交付规则、流程和经验，团队很快就会被自己过去的选择拖慢。复利工程的核心价值，就在于逼你把每次工作拆成两部分看：

- 这次功能有没有做完
- 下次同类事情会不会更简单

能同时回答好这两个问题，系统才真的在变强。
