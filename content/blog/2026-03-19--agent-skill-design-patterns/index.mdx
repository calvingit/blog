这条帖子最值得记下来的，不是又总结了五个模式，而是它先把一个更现实的变化说清楚了：当 `SKILL.md` 的目录结构、YAML frontmatter 以及 `references/`、`assets/` 这些约定，已经被 Claude Code、Gemini CLI、Cursor 等一批工具共同接受之后，格式本身就不再是主要问题。真正开始拉开效果差距的，是 Skill 里面到底怎么写。

很多人做 Skill 时还在想“字段齐不齐”“目录对不对”，但这些更像包装层。真正影响 agent 表现的，是你有没有把知识、模板、检查表、提问顺序和流程门槛放在对的位置。换句话说，Skill 的难点已经从 specification 迁到了 content design。

:::tip[核心判断]
如果说 Skill 规范解决的是“怎么打包”，那这五种模式解决的就是“怎么让 agent 少犯错、少跳步、少靠猜”。
:::

## 现在真正缺的不是格式，而是结构

原帖里有个判断我很认同：规范只定义了 Skill 的外壳，却几乎没有教你怎么组织内部逻辑。一个封装 FastAPI 约定的 Skill，和一个强制四步走的文档流水线，外面都叫 `SKILL.md`，但它们服务的问题完全不同。

所以只会照着 spec 搭目录，通常还不够。你得先判断，自己面对的到底是哪一类任务失控：是 agent 缺领域知识，还是输出结构不稳定；是评审标准容易飘，还是它老爱跳过提问和确认，直接往下生成。

五种模式的价值，就在于它们分别对准了这几类问题。

## 五种模式，分别在修什么毛病

### Tool Wrapper：把“领域知识”按需注入，而不是硬塞进系统提示词

`Tool Wrapper` 是最容易落地的一种。它不解决复杂流程，只解决一件事：当 agent 碰到某个库、某个框架、某套内部规范时，能临时加载对应知识，再按这些规则执行。

它适合的不是“帮我做完整项目”，而是“进入某个技术域时不要乱写”。比如 FastAPI 约定、React 最佳实践、数据库规范、团队内部 code style，这些内容本来就更像参考手册。把它们放进 `references/`，让 agent 在命中关键词后再加载，比把一大坨规范常驻在系统提示里更稳，也更省上下文。

我会把它理解成一种按需加载的专家模式。问题不是让 agent 变聪明，而是让它在特定场景里别偏离共识。

### Generator：当你要的不是“可能正确”，而是“每次都长得差不多”

`Generator` 针对的是另一类常见问题：agent 每次都能写出东西，但结构总在漂。今天像报告，明天像备忘录，后天又丢掉某几个必填段落。

这种时候，关键就不是补更多知识，而是把输出格式先钉住。模板放在 `assets/`，风格和质量要求放在 `references/`，再让 Skill 负责收集缺失变量并填充。重点不是提高创造力，而是降低输出方差。

技术报告、API 文档、commit message、项目方案，都很适合 `Generator`。因为这类任务真正贵的不是写一句话，而是漏字段、漏章节、漏约束。

### Reviewer：把“检查什么”和“怎么检查”拆开

`Reviewer` 是我觉得最容易被低估的一种模式。很多人写审查类 Skill，会把所有审查规则直接揉进 prompt，最后得到一个又长又脆的说明书。问题是，一旦审查标准变了，你只能回去重写整段指令。

这个模式的核心很干脆：把“检查什么”放进 checklist，把“怎么产出审查结论”留在 Skill 协议里。这样同一套审查流程可以挂不同 rubric，今天审代码质量，明天审安全规则，后天审文档规范，执行方式不变，但检查目标可以换。

这类拆分的价值不只是复用，更重要的是可维护。审查标准本来就会迭代，应该被当成独立资产管理，而不是埋在 prompt 里一起腐烂。

### Inversion：先逼 agent 问问题，再允许它动手

`Inversion` 对付的是 agent 最典型的毛病之一：信息不够时先猜，再把猜测包装成完整答案。

这个模式不是增强能力，而是反着约束行为。它要求 agent 先进入访谈状态，分阶段收集目标、用户、约束、环境和非功能要求，在必要信息补齐之前，禁止它开始设计或执行。

很多所谓“agent 很会规划”的错觉，其实只是它很会在假设上继续生成。`Inversion` 的价值，就在于把“不要脑补”从一句软提示，变成一个硬门槛。

如果任务本身变量很多、代价高，或者用户最开始给的上下文明显不足，这个模式会比直接生成靠谱得多。

### Pipeline：真正需要门禁的，不是思路，而是步骤

五种模式里最重的是 `Pipeline`。它适合的不是所有复杂任务，而是那些一旦跳步就会直接坏掉的任务。

它的重点不是把步骤写出来，而是给每一步加门禁条件。比如没有完成 API inventory，就不能进入文档装配；没有得到用户确认，就不能继续下一阶段；没有跑完质量检查，就不能交付最终产物。

很多 agent 工作流表面上也是多步骤，但实际上没有 gate，于是模型一旦嫌麻烦，就会跳过中间验证直接给结果。`Pipeline` 的存在，就是把“应该按顺序做”升级成“如果前一步没过，就不允许往下走”。

这类模式在生成文档、执行迁移、整理发布流程时尤其有用，因为这些任务失败，往往不是不会做，而是中间少做了一步检查。

## 真正有用的，不是背下五个名字，而是先选对问题类型

我读完后的最大收获，不是记住 `Tool Wrapper`、`Generator`、`Reviewer`、`Inversion`、`Pipeline` 这五个标签，而是以后写 Skill 前可以先问一句：我现在最想控制的，到底是什么？

如果要控制知识边界，用 `Tool Wrapper`；如果要控制输出形状，用 `Generator`；如果要控制审查一致性，用 `Reviewer`；如果要控制前置信息收集，用 `Inversion`；如果要控制执行顺序和校验节点，用 `Pipeline`。

这比从空白页开始瞎写 `SKILL.md` 有效得多。设计模式在这里的意义，和软件工程里很像，都是先帮你把问题分型，降低从零开始组织结构的成本。

## 模式不是互斥的，真正成熟的 Skill 往往会组合

原帖最后一点也很关键：这些模式不是互斥的。

一个 `Pipeline` 末尾可以接一个 `Reviewer` 做收尾检查；一个 `Generator` 前面可以先套一层 `Inversion`，把变量问清楚再填模板；一个 `Tool Wrapper` 也可以被放进更大的流水线里，作为某一步的领域知识加载器。

这件事背后的工程含义是，Skill 不该被理解成一段提示词，而更像一组有边界的运行时组件。你在设计的不是“怎样写一份更长的说明”，而是“怎样把知识、提问、生成、评审和门禁拆成可组合的结构件”。

如果只靠一段大 prompt 试图同时解决所有问题，最后通常两头落空：既不稳定，也不可维护。

## 如果让我压成一条实践建议

如果你最近正准备写一批 Skill，我会建议先别着急优化措辞，也别先卷 YAML。先做一件更有用的事：把你的 Skill 按失控类型分桶。

哪些是知识型问题，哪些是模板型问题，哪些是审查型问题，哪些必须先问清楚再做，哪些又必须带硬检查点。只要这个分型做对，`SKILL.md` 的骨架通常就不会太难搭；反过来，如果问题类型没分清，哪怕 frontmatter 写得再漂亮，Skill 也很容易变成一团职责混杂的 prompt。

这篇帖子表面在讲五种模式，实际上在提醒另一件更重要的事：Agent Skill 设计已经进入“内容工程”阶段了。格式越来越统一之后，真正稀缺的，不是会不会打包，而是会不会组织。
