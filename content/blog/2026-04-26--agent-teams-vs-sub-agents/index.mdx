面对复杂的 AI 任务，很多开发者的第一反应是“多加几个 Agent”。这往往是一个昂贵的误区。

**系统设计的核心不在于 Agent 的数量，而在于任务所需的协作模式。** 与其去数你需要多少个 Agent，不如先问一个更本质的问题：这个任务到底需要什么样的协调？是需要完全隔离执行的 **Sub-Agents**，还是需要实时通信的 **Agent Teams**？强行引入多智能体架构，往往只是徒增系统协调开销，甚至导致崩溃。

## 两种截然不同的协作范式

### Sub-Agents：高度隔离的“函数调用”

Sub-Agents 的本质是**隔离与并行**。你可以把它看作是一个高级的函数调用。

它是一个被高度隔离的实例，拥有自己独立的系统提示词、工具集和上下文。它的职责非常明确：接收输入，把混乱的探索过程在内部消化，最后向父级返回一个干净的信号或结果。

在 Sub-Agents 模式下，Agent 之间无法直接对话，也不能互相创建新实例。这就像是把任务分发给不同的专业外包团队——你只关心最终交付的结果，至于他们内部是如何运作的，父级 Agent 并不关心。

### Agent Teams：动态交互的“操作系统”

与 Sub-Agents 的孤立性不同，Agent Teams 更像是一个**动态运行的操作系统**。

它们的核心在于**协作**，通过共享任务层和实时通信来同步状态。比如，在一个由 Agent Teams 组成的开发环境中，前端 Agent 发现了一个接口变动，它可以立刻通知后端 Agent 进行调整。这种模式是持久的，且具有极强的交互性，适合需要高频状态同步的复杂任务。

## 任务拆分的致命误区：按角色划分

很多人在设计 Agent 架构时，习惯性地照搬人类的组织架构，按“角色”进行拆分：规划者（Planner）、开发者（Coder）、测试者（Tester）。

**这种做法会导致严重的上下文丢失。**

当任务在不同角色的 Agent 之间流转时，每一次交接都是一次信息熵增。执行者不清楚规划者的初衷，测试者也不了解执行者在写代码时面临的决策细节。这种断层会让整个系统变得极其脆弱。

## 正确的拆分逻辑：基于上下文边界

真正有效的任务拆分，不是看“谁负责干什么”，而是看“**上下文的边界在哪里**”。

如果两个任务共享着深层的背景信息和业务逻辑，就把它们留在同一个 Agent 里。**只有当上下文可以被干净地剥离、且不需要频繁的状态同步时，才应该进行拆分。**

在具体设计时，可以参考业界常用的五种架构模式：

- **提示词链（Prompt Chaining）**
- **路由（Routing）**
- **并行化（Parallelization）**
- **编排者-执行者（Orchestrator-Worker）**
- **评估者-优化者（Evaluator-Optimizer）**

如果任务本身很简单，或者不同步骤之间存在极强的依赖关系，单体大模型（单一 Agent）往往是最好的选择。盲目追求多智能体，反而会让系统在无尽的协调和通信中走向崩溃。
