面对复杂的 AI 任务，很多开发者的第一反应是“多加几个 Agent”。这往往是一个昂贵的误区。

**系统设计的核心不在于 Agent 的数量，而在于任务所需的协作模式。** 与其去数你需要多少个 Agent，不如先问一个更本质的问题：这个任务到底需要什么样的协调？

答案将决定你整个架构的走向。

Claude 风格的系统为我们提供了两种截然不同的架构路径：Sub-Agents（子智能体）和 Agent Teams（智能体团队）。它们看似相似，但解决的是完全不同的问题。

![Sub-Agents vs Agent Teams 架构对比](https://pbs.twimg.com/media/HGreTwvaMAA2pxv?format=jpg&name=900x900)

## 两种截然不同的协作范式

### Sub-Agents：高度隔离的并行执行

Sub-Agents 的本质是**隔离与并行**。你可以把它看作是一种高级的委托机制：你分配专注的工作，它返回一个干净的结果。

每个 Sub-Agent 拥有：
- 定义其角色的系统提示词
- 受限的工具集
- 完全隔离的上下文
- 单一且明确的任务目标

![Sub-Agents 工作流](https://pbs.twimg.com/media/HGret2raIAA_l3D?format=jpg&name=900x900)

当它完成任务时，它**只返回最终输出，不包含推理过程或中间步骤**。这一点非常关键：Sub-Agents 的核心价值在于**信息压缩**，而不仅仅是速度。它们负责把混乱的探索过程提纯为一个干净的信号。

**为了保持系统的可预测性，Sub-Agents 有着严格的约束：**
- 不能互相直接对话
- 不能生成新的 Agent
- 所有的信息交互必须流经父级节点

以下是一个使用伪代码（参考 `claude_agent_sdk`）实现 Sub-Agents 路由的典型例子：

```python
from claude_agent_sdk import query, ClaudeAgentOptions, AgentDefinition

async def main():
    async for message in query(
        prompt="Review the authentication module for issues",
        options=ClaudeAgentOptions(
            allowed_tools=["Read", "Grep", "Glob", "Agent"],
            agents={
                "security-reviewer": AgentDefinition(
                    description="Find vulnerabilities and security risks",
                    prompt="You are a security expert.",
                    tools=["Read", "Grep", "Glob"],
                    model="sonnet",
                ),
                "performance-optimizer": AgentDefinition(
                    description="Identify performance bottlenecks",
                    prompt="You are a performance engineer.",
                    tools=["Read", "Grep", "Glob"],
                    model="sonnet",
                ),
            },
        ),
    ):
        print(message)
```

> **核心细节**：在上述配置中，`description` 字段扮演着至关重要的“路由信号”角色。父级 Agent 通过它来决定将任务分发给哪个专职 Sub-Agent。

### Agent Teams：通过通信实现协作

与 Sub-Agents 的孤立性不同，Agent Teams 专为**协作**而生。

在这个模式下，你面对的不再是孤立的打工人，而是能够维持上下文、实时通信并动态适应的 Agent 集合。

一个典型的 Agent Team 包含：
- 一个负责分配任务和综合结果的**领导者（Lead agent）**
- 负责具体执行的**团队成员（Teammates）**
- 一个用于追踪进度和依赖关系的**共享任务层（Shared task layer）**

![Agent Teams 协作机制](https://pbs.twimg.com/media/HGrez0UaEAAX9P2?format=jpg&name=900x900)

这种机制允许真正的协调。比如，前端 Agent 发现了一个接口变动，它可以立刻通知后端 Agent 进行调整，实现状态的实时同步。

### 核心差异总结

- **Sub-Agents 侧重于执行（Execution-focused）**：孤立（Isolated）、无状态（Stateless）、一次性（One-shot）、由父级控制（Parent-controlled）。
- **Agent Teams 侧重于协作（Collaboration-focused）**：持久（Persistent）、交互式（Interactive）、共享上下文（Context-sharing）、点对点（Peer-to-peer）。

**一句话原则：** 任务独立时用 Sub-Agents；任务相互依赖时用 Agent Teams。

## 任务拆分的致命误区：按角色划分

很多人在设计多智能体架构时，习惯性地照搬人类的组织架构，按“角色”进行拆分：规划者（Planner）、开发者（Coder）、测试者（Tester）。

**这种做法会导致严重的上下文丢失。**

执行者不清楚规划者的初衷，测试者也不了解执行者在写代码时面临的决策细节。质量在每一次边界交接中不断下降。

## 正确的拆分逻辑：基于上下文边界

真正有效的任务拆分，是基于**上下文边界**的解耦。

你需要问自己：**这个任务到底需要哪些深层信息？**

![基于上下文边界的任务拆分](https://pbs.twimg.com/media/HGre4EaawAECVXq?format=jpg&name=900x900)

如果两个任务共享着深层的背景信息，就把它们留在同一个 Agent 里。**只有当上下文可以被干净地剥离时，才应该进行拆分。**

在具体设计时，可以参考业界常用的五种架构模式：

![五种核心架构模式](https://pbs.twimg.com/media/HGre8HpbkAAkYSx?format=jpg&name=900x900)

1. **提示词链（Prompt Chaining）** —— 顺序执行的流水线
2. **路由（Routing）** —— 将特定任务分发给最合适的 Agent
3. **并行化（Parallelization）** —— 同时运行相互独立的工作
4. **编排者-执行者（Orchestrator-Worker）** —— 由单一中心 Agent 进行任务委派
5. **评估者-优化者（Evaluator-Optimizer）** —— 持续的生成与迭代优化

## 什么时候不该使用多智能体？

很多时候，一个强大的单体大模型（Single Agent）就已经足够了。与其盲目增加复杂度，不如遵循以下判断标准：

**✅ 适合引入多智能体系统的场景：**
- 你需要严格的上下文隔离
- 存在可以并行处理的任务
- 任务需要极度专门化（Specialization）的工具或提示词

**❌ 应避免引入多智能体系统的场景：**
- 任务之间存在极强的相互依赖
- 协调和通信的开销过大
- 任务本身非常简单

**最终原则：围绕“上下文边界”进行设计，而不是“角色”。从最简单的架构开始，只在真正需要时才引入复杂性。**
