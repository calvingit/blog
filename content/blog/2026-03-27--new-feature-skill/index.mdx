## 背景

过去一段时间，我们一直在想办法把 AI 更稳地接进多客 App 的日常开发。先看局部任务，这条路是走得通的。让模型改一个页面、补几个接口，已经比完全放开靠谱不少。

但需求一旦从“补一个点”变成“做一个完整功能”，老问题就又冒出来了。

在移动端开发里，一个完整功能很少只是改一个文件。它常常会同时碰 API 文档、Model、Repository、ViewModel、Page、测试、路由、DI，有时还得兼容旧逻辑或者补国际化。AI 当然能写其中一段，但只要没人盯顺序和边界，它很快就会在“先做什么、后做什么、哪些该交给谁做”上乱套。

所以我们做 `new-feature`，想补的其实就是编排这一层。先把完整功能开发里的分析、拆分、先后顺序和边界约束定下来，再让 AI 去写。这样出来的结果就不再是一堆松散代码，而更接近一次按团队习惯完成的跨层协作。

这篇文章不打算讲某个业务细节，也不是要分享什么 Flutter 小技巧。我们更想回头看一件事：一段原本很靠经验的 Vibe Coding 流程，是怎么一点点收成一套能复用、能评估、也还能继续打磨的 Skill 体系的。

## 当前 Vibe Coding 存在的问题

如果只看单次对话，Vibe Coding 很容易给人一种“差不多能用了”的错觉。模型能理解需求，也能给出一套看起来像样的建议，甚至还能一次吐出不少代码。但真把它放进多客 App 这种有明确分层、路由体系、依赖注入和文档规范的 Flutter 项目里，问题很快就会露出来。

这里有一个很容易被忽略的前提：在做 `new-feature` 之前，我们并不是一张白纸。项目里已经有一批能解决局部问题的子 Skill，大致如下：

| Skill                 | 用途                                              |
| --------------------- | ------------------------------------------------- |
| `generate-dart-model` | 根据 JSON 或接口响应生成 Dart Model               |
| `new-repository`      | 创建 Repository、Mock，以及最小 DI 接入           |
| `new-viewmodel`       | 抽离页面级业务状态，创建 ChangeNotifier ViewModel |
| `new-page`            | 新建页面模块、补页面结构和路由接入                |
| `new-test`            | 按项目测试规范补测试文件                          |
| `new-spec`            | 收尾时归档 spec 文档                              |
| `dokoup-refactor`     | 处理新旧代码兼容和旧逻辑边界                      |
| `duoke-components`    | 约束 Flutter UI 使用 DK 组件和设计 token          |

也就是说，团队当时缺的并不是“会提炼 Repository 的 Skill”或者“会写页面的 Skill”，这些能力本来就有。

问题在于，这些 Skill 大多只解决单点任务。一旦需求同时跨了 API 文档、数据层、状态层、页面层和测试层，它们并不会自己长成一条稳定流程。模型知道仓库里“有这些 Skill”，但还是会卡在下面这些地方：

第一个问题是任务边界模糊。一个需求里同时出现 Page、Repository、ViewModel、Test 时，模型很容易把它们搅在一起。它知道这些都“要做”，但分不清哪些属于分析阶段，哪些属于设计阶段，哪些是单领域执行，哪些必须等前置结果稳定后才能开工。最后给出来的东西看着挺全，实际上没有拆开。

第二个问题是跨层依赖顺序不稳定。移动端功能开发不是所有模块都能并行。Model 没定下来，Repository 的返回类型就定不住；Repository 的接口没定下来，ViewModel 的状态设计就会反复改；ViewModel 还在飘，Page 层的 `Consumer<T>` 绑定也会跟着重写。可一旦进入自由发挥模式，模型特别容易先写页面，再回头补数据层。看上去很勤快，实际只是把返工提前了，Token 和时间一起浪费。

第三个问题是共享文件的风险经常被忽略。多客 App 项目里有几类文件天然就是共享的，比如 `lib/core/dependency_inject.dart`、`lib/router/go_router_registry.dart`、`docs/api/README.md`。它们不是业务文件，但通常比普通页面更难动。问题不在于模型改不了，而在于它不会主动把这些文件当成风险点。于是看起来像个轻任务，真做起来已经变成跨模块协作。

第四个问题是“会分析，不会落地”。模型经常能说出一些方向没错的话，比如“建议先确定数据结构”“建议按分层设计”，但这些话不会自己变成可执行调度。它可能方向说对了，却没有给出真正能用的子 Skill 清单，没有批次划分，没有共享文件的写入边界，也没有能直接发给 sub-agent 的 prompt。最后分析和执行还是断开的，主 agent 只能现场补决策。

第五个问题是没有评估闭环。没有 benchmark 的时候，我们只能靠“这次看起来还行”来判断 Skill 有没有价值。局部任务里还能凑合，一旦任务跨层，主观判断就会飘得很厉害。你今天觉得这个 Skill 很聪明，明天换一个稍微不同的需求，它可能又开始乱调度。如果没有 with/without 对比，没有典型用例，也没有失败案例回放，所谓“优化 Skill”最后很容易变成凭感觉改 prompt。

这些问题放在一起，结论其实很直接：AI 不是不会写代码，我们只是还没把它放进一个合格的协作流程里。

## 解决方案

往下走，大致有三种办法：

1. 做一个单体大 Skill，把完整功能开发里可能碰到的规则全塞进去，从 API 文档一路管到 Page 和测试。这个思路最直观，但代价也最大，相当于把 `new-repository`、`new-page`、`new-viewmodel`、`new-test` 这些已有 Skill 的规则又写了一遍。规则会越堆越厚，Skill 之间一旦重叠，后面维护会很难受。

2. 继续放任自由调用子 Skill。也就是不做总控，只让模型自己判断“这个任务该不该用 `new-page`、要不要配 `new-repository`”。这个方式最灵活，但也最不稳。模型不会天然知道“Page 应该等 Repository 和 ViewModel 稳定之后再做”，也不会天然知道 `docs/api/README.md` 和 `lib/l10n/*.arb` 为什么比普通业务文件更敏感。

3. 也是我们最后选的方案，单独补一个编排层：由 `new-feature` 负责分析需求、判断复杂度、决定子 Skill 清单、给出执行批次和共享风险提示；真正的实现还是交给已有子 Skill 去做。

这个方案成立，有两个前提。第一，主 agent 负责判断和设计，不一头扎进所有细节实现。第二，子 Skill 只做单领域执行，不重新做需求分析。这样分工才会稳：主 agent 决定“做什么、先后顺序是什么、哪些东西不能并发改”；子 Skill 负责在各自领域里按项目规范做对。

`new-feature` 做的事很明确：先把完整功能开发里的协作顺序搭起来。

## Skill 设计思路

### 1. 用 9 个维度把“完整功能开发”拆开

`new-feature` 的第一层设计先不急着做调度，我们先把一个看似完整的大需求拆成几个稳定维度。最后固定下来的，是这 9 个维度：

1. API 文档
2. 新旧代码兼容
3. Dart Model
4. Repository
5. ViewModel
6. 新页面/路由
7. UI 组件
8. 测试
9. 需求归档

这么拆，说白了就是逼着主 agent 先把“这次到底会碰哪些领域”说清楚。以前模型经常直接跳进实现建议，却没先回答“哪些需要做、哪些不需要、为什么”。先做完这 9 个维度判断，需求才算真正拆开，后面的批次编排和复杂度判断才有稳定输入。

这里有个后来被反复验证的细节：`new-feature` 不把“API 文档”当成独立 Skill。它会把这件事识别成一个领域，但执行上还是交给主 agent 或通用 worker，而不是硬造一个并不存在的子 Skill。后面我们修 sub-agent prompt 时，这个判断帮了很大忙。

### 2. 复杂度门控不能只看“命中了几个维度”

只做 9 个维度分析还不够，因为复杂度并不只看维度多少。我们一开始最容易犯的错误，就是拿“命中了 4 个以上维度就先写 plan，否则可以直接做”这种简单规则来判断复杂度。后来发现，这个规则没有我们想的那么稳。

问题在于，有些需求维度不高，但只要碰到共享文件，复杂度就会立刻上来。比如只给页面顶部加一个风险提示条，看起来只是 UI 加测试，但如果会改 `lib/l10n/*.arb`，那就已经不是普通页面改动了。再比如当前阶段只做 API 文档、Model 和 Repository，看起来只有 3 个维度，但只要涉及 `docs/api/README.md`，同时又横跨 `docs/api/` 和 `lib/`，就不该再被算成轻任务。

所以我们最后把复杂度门控收紧成固定判定顺序：

1. 先看有没有需求歧义或多种方案。
2. 再看是不是跨模块改动。
3. 再看触发维度是否达到高复杂度。
4. 只有前三项都不命中，才允许跳过 plan。

后来我们把几类共享文件直接提升成跨模块硬判定：`dependency_inject.dart`、`go_router_registry.dart`、`docs/api/README.md`、`lib/l10n/*.arb`，以及同时横跨 `docs/api/` 和 `lib/` 的改动。这样做主要是为了堵住一个常见漏洞：模型很爱拿“维度不高”来掩盖真正的协作风险。

### 3. 子 Skill：不解决一切，它们只解决自己的问题

在 `new-feature` 里，子 Skill 不是按“哪个顺手就先用哪个”来组合，而是按职责排。这个设计看起来很朴素，但对稳定性影响很大。

最后沉淀下来的子 Skill 关系，大致是这样：

| 领域           | 对应能力              | 责任边界                                |
| -------------- | --------------------- | --------------------------------------- |
| 新旧代码兼容   | `dokoup-refactor`     | 分析旧代码边界和兼容策略                |
| 数据结构       | `generate-dart-model` | 生成 Model，稳定字段结构                |
| 数据访问       | `new-repository`      | 建 Repository、Mock、DI 最小接入        |
| 页面级业务状态 | `new-viewmodel`       | 提取 ChangeNotifier 和状态方法          |
| 页面/路由      | `new-page`            | 页面骨架、路由接入、页面结构            |
| UI 规范        | `duoke-components`    | 作为 UI 阶段的贯穿约束，而不是独立步骤  |
| 测试           | `new-test`            | 为 Repository / ViewModel / Page 补测试 |
| 收尾归档       | `new-spec`            | 功能完成后归档 spec                     |

这里有个很容易漏掉的点：`duoke-components` 不是单独的一步。它更像是 Page/UI 阶段的全程约束。如果把它当成独立串行步骤，调度会很别扭，好像 UI 规范可以先做完，再去做页面。实际并不是这么回事，它应该贯穿在 `new-page` 以及相关 UI 产出里。

### 4. sub-agent 编排：执行链看的是依赖关系，不是并行越多越好

`new-feature` 的执行链最后被分成 4 个批次：

- 批次 A：API 文档、`dokoup-refactor`
- 批次 B：Model → Repository → ViewModel → Page
- 批次 C：测试并行
- 批次 D：`new-spec`

这么排不是为了好看，而是因为批次 B 的依赖关系确实很硬。Model 不稳定时，Repository 的方法签名会漂；Repository 不稳定时，ViewModel 的依赖字段和 public methods 也会漂；ViewModel 不稳定时，Page 的消费方式和初始加载逻辑又得跟着改。这个链条在 Flutter 场景里尤其明显，因为 `ChangeNotifier`、`Consumer<T>` 和 Page 层数据绑定之间本来就绑得很紧。

测试之所以放到批次 C，是因为只有当前置接口稳定后，测试才能真正独立并行。如果在 Repository 还没定的时候就去写测试，得到的通常不是提前验证，而是提前返工。

所以 `new-feature` 这层编排最重要的工作，其实就是把“谁必须等谁”定清楚。

## Sub-agent 设计：从“能派发”到“真正可派发”

`new-feature` 做到这里以后，我们很快又撞上一个问题：就算批次有了，sub-agent 也不一定能稳定干活。

最开始我们以为，只要把 prompt 模板写出来，sub-agent 就能用了。后来发现，这其实只解决了“看起来像个任务”的问题，没有解决“它到底能不能执行”的问题。

一个真正可派发的 sub-agent prompt，至少要包含 4 个要素：

1. 所用 Skill 路径
2. 完整上下文，也就是 HANDOFF 里的字段
3. 文件所有权与禁改范围
4. 固定返回格式

少任何一个，主 agent 后面都会补得很痛苦。

我们这次最典型的教训就出在 `references/sub-agent-prompts.md`。一开始模板已经不算短了，但真拿出来审，问题还是一堆。比如 API 文档那一步最初被写成了 sub-agent prompt，但它其实没有对应的独立 Skill，更准确的说法应该是“通用文档 worker 任务”，而不是 Skill-backed sub-agent。再比如 `new-repository` 那步，最开始只给了 Model 和接口信息，却没给 `remote/local`、`global/page`、`DI scope` 这些关键决策。看着像在派任务，实际上 sub-agent 根本不知道 Repository 该建在哪里、该不该碰 DI。`new-page` 也一样，最初默认就是“创建页面并注册路由”，但 `new-page` 本身区分 `Tab 内页面` 和 `子路由页面`，前者本来就不该注册路由。

这件事也让我们重新看了一遍 prompt 的角色。sub-agent 提示词更像是“写给执行系统的任务协议”。它不光要说明做什么，还得把输入、可改文件、禁改范围、缺上下文时怎么返回写清楚。只有这样，sub-agent 才不会变成“听懂一点就开改”，而会是一个真正可预测的执行节点。

## prompt 编写：为什么 HANDOFF 比漂亮措辞更重要

在主 agent 和 sub-agent 协作里，prompt 写得像不像人话，其实没那么重要。真正关键的是，上下文是不是已经结构化到足够支撑下一步执行。我们后来在 `sub-agent-prompts.md` 里补齐的，也不是更漂亮的文档，而是一组协议字段。

比如在 Model 阶段，主 agent 至少要把 `api_doc_path`、`api_response_fields`、目标输出目录和目标文件路径交给 `generate-dart-model`。在 Repository 阶段，至少要补齐 `kind`、`scope`、`di_scope`、目标文件路径和 API 文档位置。在 Page 阶段，至少要补齐 `page_type`、目标页面文件、路由文件和 ViewModel 的稳定接口。如果这些字段都没有，那不是 sub-agent 发挥失常，是任务本身就没定义完整。

除了上下文，文件所有权也是 prompt 里必须显式写出来的。后来我们收紧模板时，统一加上了“你不是唯一在改这个仓库的人，不要回滚或覆盖他人修改”“只允许修改哪些文件”“如果缺少关键上下文，直接返回 BLOCKER”这类约束。它们看起来不花哨，但这是让多 worker 协作稳定起来的前提。

说到底，我们后来对 prompt 编写的理解也变了：主 agent 先要把一个最小可执行单元声明清楚。输入、边界和返回格式先钉住，模型的生成能力才真正用得上。

## 评估 Skill

如果没有 eval，`new-feature` 很可能最后只会停在“大家觉得思路不错”的阶段。我们这次做评估时，最坚持的一件事，就是一定要用 Claude 官方的 `skill-creator` Skill 做 `with_skill` / `without_skill` benchmark 对比，而不是只看用了 Skill 之后的单边表现。

评估用例的设计也不是随便拼几个 prompt。我们专门覆盖了几类最容易出问题的场景：

- 复杂全链路场景：同时涉及 API 文档、Model、Repository、ViewModel、Page、测试、旧逻辑兼容、路由。
- 轻量 UI 场景：只改一个页面 UI，加一个 widget test。
- 跨模块低维场景：维度不多，但会碰路由、DI 这类共享文件。
- ARB 场景：看似只是 UI 和测试，但会改 `lib/l10n/*.arb`。
- API 文档场景：只做文档、Model、Repository，不做 Page、ViewModel 和测试。

对应的评估标准也不是“有没有提到某个词”。我们更关心的是它是否做出了正确的判断：该不该先写 plan，哪些子 Skill 需要用，哪些不需要，批次顺序是否稳定，共享风险文件有没有点名，`duoke-components` 是否被当作贯穿约束，而不是独立串行步骤。

评估过程里踩过的坑也挺典型。

第一类是基础问题。比如一开始 `SKILL.md` 的 `description` 就写错了，写成了文档说明，没有用 "use when" 这种触发式语句，结果 Skill 很难被触发。这不是文档小瑕疵，而是会直接影响触发和评估。

第二类是“规则写在正文里，但模型不一定稳定执行”。后来我们发现，只把复杂度门控和共享风险文件写进 Skill 正文还不够，模型在某些场景下还是会退回到更简单的判断。

第三类是评估断言本身。断言太宽时，baseline 也能拿高分；断言太死时，with_skill 和 baseline 又会一起被压下去，看着更严格了，区分度却没有真的拉开。

其中有两个 case 特别有代表性。`eval-0` 是复杂全链路场景，`eval-5` 是 API 文档 + Model + Repository 场景。后期我们专门收紧了这两个 case 的断言，想把它们和 baseline 真正拉开。结果最后一轮里，`eval-0` 和 `eval-5` 还是打平，暴露出来的不是 Skill 本体退化，而是评估逻辑仍然太依赖固定措辞，没有抓住真正的结构性信号。这件事也提醒了我们：benchmark 自己也是要反复设计和迭代的。

## 成果

下面每一组数字，前者都是 **with_skill** 的聚合通过率，后者都是 **without_skill** 的 baseline 聚合通过率。它们反映的是：用了 `new-feature` 之后，模型在整组评测用例里有多稳定地做出了我们想要的编排判断。这不是业务指标，也不是某一次回答的单次得分。

在这个口径下，我们先后看到过几组关键结果：

- 初版对比：with_skill `80.7%`，without_skill `44.3%`
- 扩充用例后：with_skill `72.3%`，without_skill `27.7%`
- 收紧复杂度门控后的高点：with_skill `95.8%`，without_skill `68.0%`
- 收紧断言后的复验：with_skill `87.0%`，without_skill `70.2%`

这些数字不能直接横向比较，因为每一轮的用例集和断言都在变。更合理的看法，是把它们当成四个阶段给出的信号。

第一阶段先回答了一个最基本的问题：`new-feature` 值不值得做。初版对比里，with_skill 相比 baseline 已经有比较明显的优势，说明“先分析、再编排、最后执行”这条路是走得通的。

第二阶段是在扩充用例后主动给自己加压。当我们把 ARB 场景、局部 ViewModel 场景、API 文档场景加进去后，with_skill 的通过率降到了 `72.3%`，但 baseline 也只有 `27.7%`。这更像是在提醒我们，前一阶段并不是 Skill 已经做完了，而是测试集还不够严。边界 case 一加进来，很多原本看起来稳定的规则就开始晃。

第三阶段最像一次正经的工程修正。我们把 `docs/api/README.md`、`lib/l10n/*.arb` 这类共享文件正式提升成跨模块硬判定，同时强制主 agent 在分析结果里单列共享写入风险文件。收紧之后，benchmark 一度跑到了 `95.8% vs 68.0%`。这里真正起作用的，是跨模块规则终于从建议变成了硬约束。

第四阶段则提醒我们别太迷信高分。为了验证评估是不是够严格，我们又收紧了复杂场景和 API 文档场景的断言，最终整体验证回到 `87.0% vs 70.2%`。表面上看 with_skill 还是领先，但 `eval-0` 和 `eval-5` 没有被真正拉开。这个结果很关键，因为它说明 benchmark 不只是拿来“证明自己厉害”的，也能拿来暴露评估设计本身的问题。

真要说这次复盘里最值钱的地方，是我们终于能把失败讲明白了。比如 `description` 写错为什么会直接卡住 Skill；“API 文档不是独立 Skill”为什么必须写进断言；有些断言为什么看起来更严格，却没有真的把区分度拉开。把这些问题讲透，后面的迭代才不会变成盲改。

## 总结

回头看这次 `new-feature` 的创建过程，最后留下来的不只是一个 Skill 文件夹，还有一套我们对 AI 协作工程化的认识。

第一，适合 Skill 化的不是一次性的业务知识，而是稳定的决策模式。今天是订单筛选页，明天可能是审批记录页，具体业务会变，但“完整功能开发要先拆维度、再判复杂度、再做批次编排”这种模式可以反复用。

第二，主 agent 和 sub-agent 的边界必须写成制度，不能靠默契。主 agent 负责分析、判断、设计和调度；sub-agent 负责在明确输入和文件边界下执行单领域任务。只要这条边界一模糊，后面无论 Skill 写得多长，都很容易重新滑回 Vibe Coding。

第三，benchmark 不是验收时顺手带一下的附属品，它本来就是 Skill 设计的一部分。没有评估，Skill 只能靠感觉优化；有了评估，才知道哪些规则真的有效，哪些只是听起来有道理。

第四，Skill 的终点不只是“能触发”，而是“能稳定产出正确的协作行为”。如果一个 Skill 只能在最理想的 case 下表现不错，那它更像演示稿，不像工程能力。真正有用的 Skill，应该能在复杂场景、边界场景和失败场景里都保持可预测。

如果顺着这条路继续往前走，我们接下来还会做三件事。第一，继续优化 eval 的区分度，尤其是复杂场景里的结构性断言。第二，评估是否有必要把某些稳定模式继续拆成独立 Skill，比如 API 文档类工作流。第三，把 prompt 和 HANDOFF 模板继续产品化，让主 agent 在派发 sub-agent 时少一点手工补充。

`new-feature` 这次复盘最重要的收获，也不只是“我们又多了一个 Skill”。我们开始把 AI 写代码这件事，从一次次临场发挥，往可设计、可编排、可评估的团队能力上推。
