如果只把 Skill 当成 `SKILL.md`，很容易掉进一种错觉：结构写对了，说明也写全了，质量应该差不多了。

但真正决定 Skill 能不能进生产环境的，往往不是“写出来没有”，而是“评出来没有”。没有评测，Skill 迭代基本只能靠体感；一旦模型版本变了、任务分布变了、上下文长度变了，之前那份“感觉还行”的 Skill 很快就会失真。

这篇文章里最值得保留的部分，不是 Skill 的基础概念，而是作者把 Skill 评测拆成了一套可以重复执行的工程流程。它的核心结论很直接：

1. Skill 不是提示词资产，而是可回归的工程资产。
2. 判断 Skill 是否有效，必须拿基线做对照，而不是只看单次输出。
3. Skill 优化不是持续加规则，而是持续缩小“任务目标”和“执行行为”之间的偏差。

## 为什么 Skill 评测比“写对格式”更重要

Skill 的价值通常体现在三件事上：

1. 让 Agent 更稳定地遵循你的流程。
2. 降低任务执行中的漏项、误判和无效步骤。
3. 在相同任务上，拿到更可预测的输出质量。

这三件事如果不做对照实验，其实都很难证明。尤其是团队里已经同时存在底模增强、工具接入、上下文补充这几类变化时，单看最终答案，你很难分清到底是谁起了作用。

所以更可靠的做法不是问“这个 Skill 看起来好不好”，而是问四个更具体的问题：

1. 开启 Skill 之后，输出质量相比基线到底提升了什么。
2. 提升是否稳定，还是只在少数示例里偶然出现。
3. 质量提升是否伴随不可接受的 token 或时延成本。
4. 这次修改到底修复了通用问题，还是只是把一个 case 调通了。

## 一套够用的评测闭环

原文把评测流程拆成了五步，我更建议按一个更紧凑的四段闭环去理解：建基线、跑实验、做评分、再迭代。

### 1. 先定义基线，不然“提升”没有参照物

Skill 评测里最常见的问题，不是不会写评分标准，而是没有选好对照组。

对照组通常有两种：

1. 没有 Skill 的裸跑结果，用来回答“这个 Skill 值不值得存在”。
2. 旧版本 Skill，用来回答“这次改动值不值得保留”。

这一步不能省。因为很多改动看上去让指令更完整了，实际只是把流程写长了；如果没有基线，你很容易把“更啰嗦”误判成“更稳定”。

:::tip[建议]
新建 Skill 时，优先和“无 Skill”对比；优化已有 Skill 时，优先和“上一版 Skill”对比。两种问题不要混在一轮实验里。
:::

### 2. 用评测集覆盖真实任务，而不是挑最好看的例子

评测用例的目标不是展示 Skill 成功，而是暴露 Skill 的边界。

一个够用的评测集，至少应该覆盖：

1. 正常任务：Skill 最主要的目标场景。
2. 边界任务：信息不完整、指令带歧义、上下文较长的场景。
3. 干扰任务：容易误触发或引发错误流程的相邻任务。

如果评测集只包含“最适合 Skill 发挥”的任务，那跑出来的分数基本没有决策价值。原文里反复强调的其实也是这一点：评测不是为了证明自己写得对，而是为了尽快找到 Skill 在哪里不对。

### 3. 评分要同时看结果和过程

很多团队只评最终输出，这样会漏掉一类很关键的问题：结果看上去能交差，但执行路径已经歪了。

比较稳妥的方式，是把评分拆成两层。

第一层看结果质量：

1. 任务是否完成。
2. 关键要求是否满足。
3. 有没有明显漏项、误判或幻觉。

第二层看执行过程：

1. Agent 有没有按预期流程使用工具。
2. 是否出现大量无效步骤或重复劳动。
3. token 消耗、响应时间是否还能接受。

原文给出的一个很实用判断是：如果质量提升是靠明显更高的成本换来的，那这次优化未必成立。Skill 不是离线 prompt 比赛，它最终要跑在真实环境里。

## 评测之后，真正决定上限的是“怎么改”

跑完分数只是开始。更难的是看到问题之后，怎么改 Skill 才不会越改越窄。

### 原则一：从反馈里抽通用规律，不要围着单个 case 打补丁

如果某次改动只修好了测试用例 B，却让测试用例 A 退步，那往往不是你找到了规律，而是你开始过拟合。

Skill 最终面对的是真实任务分布，不是手头那几个样例。评测集的价值在于提供反馈，不在于让你把它们全部“背下来”。

### 原则二：指令越长不代表越稳，很多时候恰好相反

原文里有个判断我很认同：冗余指令不只是浪费 token，还会分散模型注意力。

这意味着优化 Skill 时，先问的不是“还要不要再加一条 MUST”，而是：

1. 现有哪条指令在制造无效步骤。
2. 哪些约束只是重复表达，没有增加判别力。
3. 哪些规则更适合放进脚本，而不是继续留在自然语言里。

如果 Agent 在多个用例里反复造同一类辅助脚本、做同一类预处理，那通常说明这部分已经适合沉到 `scripts/`，不应该每次都让模型现场发挥。

### 原则三：解释 why，通常比堆 MUST 更有效

这也是 Skill 写作里很容易被忽略的一点。

很多人一发现执行不稳定，就开始把 `ALWAYS`、`NEVER`、`MUST` 写满全文。短期看像是加了约束，长期看往往会让 Skill 变得僵硬，而且一旦场景变化，模型更容易机械执行。

更有效的做法是写清楚原因：

1. 为什么这一步必须先做。
2. 为什么某类数据源更可信。
3. 为什么某种失败模式需要提前规避。

只在真正不可违反的边界上使用硬约束，Skill 的泛化能力通常会更好。

## 一个很有代表性的案例：Code Review Skill 为什么会出现代际差异

原文里最有启发的一段，是拿代码审查 Skill 做对比。

第一版其实是很多团队都会写出来的样子：一份直白的 Markdown 指令，列出审查维度和注意事项。它不是不能用，但问题也很典型：

1. 输出质量波动大。
2. 容易漏掉真正重要的问题。
3. 有时又会盯着细枝末节不放。
4. `git diff` 一长，上下文就容易被污染。

第二版的改法不是继续往提示词里加规则，而是直接改执行架构：

1. 把一次 review 拆成总览、分维度审查、交叉验证、去重汇总、最终报告几个阶段。
2. 让子 Agent 各自只拿一个文件的 diff 和源码，减少跨文件干扰。
3. 为每个阶段定义明确的输入输出契约。
4. 把任务是否完成做成可检查状态，单个子任务失败时允许重跑。

这个案例说明了一件很重要的事：有些 Skill 问题根本不是“提示词不够细”，而是任务拆分方式不对。评测的价值，正是在于逼你承认这一点。

原文提到，两版方案在相同的 20 个 PR 上做了对照评测，第二版在输出质量、覆盖率和误报率上都有明显提升。即使没有展开所有分数细节，这个实验设计本身已经足够有参考价值。

## 我会怎样把这套方法落到自己的 Skill 仓库里

如果现在要给团队里的 Skill 建一个最小可用的评测体系，我会按下面这个顺序做：

1. 先为每个核心 Skill 维护一组小而稳定的评测任务，不追求多，先覆盖主路径和高风险边界。
2. 每次修改 Skill，都同时跑实验组和对照组，确保结论来自比较而不是印象。
3. 评分时把“结果质量”和“执行成本”一起记录，避免出现高分但不可用的版本。
4. 对重复出现的问题，优先考虑改任务拆分、脚本化处理和输入输出契约，而不是继续堆指令。

这套做法的门槛其实没有想象中高。难点不在工具，而在团队是否愿意接受一个事实：Skill 和代码一样，必须靠回归测试说话。

## 结语

Skill 基础知识当然重要，但真正把 Skill 拉开差距的，通常不是“会不会写”，而是“会不会评”。

没有评测，Skill 很容易停留在经验层面；有了对照实验、评分标准和迭代闭环之后，它才会从一次性的 prompt 资产，变成可以持续维护的工程资产。

如果你已经在写 Skill，下一步最值得补的通常不是更多模板，而是一套能回答这句话的评测体系：

“这次改动，到底让它变好了多少？”
