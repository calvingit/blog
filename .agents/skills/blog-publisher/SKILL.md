# Blog Publisher

从 URL 抓取文章内容，整理成中文技术博客并发布到博客仓库。

发布前请先阅读 [references/blog-conventions.md](references/blog-conventions.md)，其中包含 frontmatter schema、有效 tags、目录命名规则、写作风格要求、校验清单与 commit 格式。

如果来源是英文或混合语言内容，再读取 [references/glossary-en-zh-tech.md](references/glossary-en-zh-tech.md) 作为默认术语种子表。

---

## 工作流程

### Step 0 — 确认项目根目录

```bash
git rev-parse --show-toplevel
```

如果命令失败（不在 git 仓库中），停止执行并提示用户先切换到博客项目目录。

---

### Step 1 — 抓取文章内容

抓取用户提供的 URL，提取以下内容：

- 文章标题、作者、发布时间、原始链接
- 文章核心论点与写作动机
- 关键概念、原理与技术细节
- 值得保留的实践经验、代码示例或工程建议
- 需要保持原样或严格受控的实体：术语、专有名词、数字、单位、代码、引用

**抓取工具优先级与降级策略：**

| 优先级    | 说明                        | 工具                                                                          |
| --------- | --------------------------- | ----------------------------------------------------------------------------- |
| 1（默认） | 所有 URL 一律先走主通道     | `markdown.new` — `POST https://markdown.new/` 或 `https://markdown.new/<url>` |
| 2（回退） | `markdown.new` 返回空或报错 | `chrome-devtools`                                                             |
| 3（兜底） | `chrome-devtools` 仍然失败  | `agent-browser`                                                               |

禁止使用 `fetch` 作为抓取路径。不要因为站点类型不同而改变主流程，统一先尝试 `markdown.new`。

**微信公众号经验：**

- 微信文章经常出现 `markdown.new` 超时、空响应或结构不完整，此时不要反复重试主通道，直接回退到 `chrome-devtools`
- 使用 `chrome-devtools` 时，优先读取页面快照或 DOM 文本，提取标题、作者、日期和正文分节
- 如果微信正文已经能在浏览器里完整展开，就不要再换抓取路径，避免二次引入噪声

**Medium 会员文章（Member-only story）特殊处理：**

- 如果原始链接是 `medium.com` 且页面显示 `Member-only story`，先把原始 Medium URL 直接拼接到 `https://freedium-mirror.cfd/` 后面，再把这个 Freedium 链接交给 `markdown.new`。
- 示例：
  - 原文：`https://alirezarezvani.medium.com/10-claude-code-commands-that-cut-my-dev-time-60-a-practical-guide-60036faed17f`
  - 镜像：`https://freedium-mirror.cfd/https://alirezarezvani.medium.com/10-claude-code-commands-that-cut-my-dev-time-60-a-practical-guide-60036faed17f`
- 抓取顺序固定为：
  1. 先尝试 `markdown.new(<freedium_url>)`
  2. 若失败，回退到 `chrome-devtools`
  3. 仍失败，再回退到 `agent-browser`
- 引用说明中的链接继续保留**原始 Medium 链接**，不要替换为 Freedium 镜像链接。

**Reddit / 论坛讨论帖特殊处理：**

- 如果输入来源是 Reddit、Hacker News、论坛长帖、GitHub issue 讨论串或评论串，不要只抓原帖正文；默认拆成“原帖观点 + 评论区观点”两层材料
- 抓取评论区时，不要求穷举所有回复，优先覆盖三类代表性声音：
  1. 赞同原帖并补充工程经验的评论
  2. 明确反对原帖结论或指出风险的评论
  3. 对适用边界、安全约束、性能成本提出补充的评论
- 写作时必须区分“原帖作者的结论”和“评论区提出的判断”，不要把评论区观点误写成原帖原意
- 如果评论区信息价值明显高于原帖正文，文章角度应优先写成“这场讨论真正暴露了什么问题”，而不是“楼主原文摘要”

如果 `markdown.new`、`chrome-devtools`、`agent-browser` 都失败，整个流程立即终止，并明确报告失败原因，不要继续进入写作阶段。

**检测到未安装`agent-browser`时，自行安装并重试抓取**

```bash
npm install -g agent-browser
agent-browser install  # Download Chromium
```

### Step 1.5 — 生成事实稿、语义分析卡片与最小术语表

默认按“个人阅读 / 文章整理”场景处理，不直接进入成文阶段，先做一层轻量整理。

先整理一份**事实稿**，只保留可验证信息，不做文风润色：

1. 原文主论点与写作动机
2. 支撑主论点的关键事实
3. 关键代码、配置、命令或数据点
4. 限制条件、适用边界、反例或 caveats

如果来源本身是讨论帖、评论串或 forum thread，再额外整理一份**讨论型事实稿**：

1. 原帖核心论点
2. 原帖支撑论点的关键事实
3. 评论区主要赞同点
4. 评论区主要反对点
5. 评论区补充的限制条件、反例或风险
6. 你最终认可或保留的判断

再整理一张**语义分析卡片**，只服务后续改写与质检，不直接进入正文：

1. 作者是谁，立场是什么
2. 这篇内容在回应什么问题或争论
3. 作者真正想推动读者接受的判断是什么
4. 原文依赖了哪些隐含前提
5. 原文语气更接近技术说明、评论、倡议、观察还是经验总结
6. 哪些比喻、习语、情绪化用词或隐含意思不适合直译

如果遇到明显的比喻、习语或带情绪色彩的表达，额外记一条简短映射，格式建议：

```text
原表达 -> 实际想表达的意思 -> 处理策略（直说 / 换中文说法 / 保留原意象）
```

先读取默认术语种子表 `references/glossary-en-zh-tech.md`，再整理当前文章的**最小术语表**。最小术语表优先收录 10-20 个高价值术语：

- 默认术语表里已收录、且本篇会反复出现的高风险术语
- 当前文章特有的垂直领域术语
- 容易误译且出错代价高的词
- 同一词在本文不同语境下可能漂移的词

默认不要收录下面几类：

- 已经约定俗成、几乎不会翻错的常见术语
- 品牌名、产品名、项目名
- 命令、代码标识符、库名、类名、函数名

如果默认术语表与当前文章语境冲突，以当前文章语境为准。

格式建议：

```text
alignment -> 对齐
grounding -> 锚定
tool calling -> 工具调用
```

如果原文本身是中文，可以跳过翻译动作，但事实稿和术语表仍然要做，用于后续写作与一致性检查。

### Step 2 — 撰写 MDX 博客文章

以**学习者视角**组织内容，优先回答「为什么重要」，再讲「是什么」，最后讲「怎么用」。不要直接翻译，要**重构与提炼**。

对于外文或混合语言来源，默认采用轻量的两轮整理流程：

1. **第一轮：忠实信息稿**
   - 只追求信息完整、事实不漂移，不追求文风
   - 保留代码块语言、数字、单位、命令、引用、专名和关键编号
   - 可以暂时保留原文信息顺序，但不要把原始章节映射当成最终结构
   - 严格遵守术语表；不确定的术语宁可保留原文并显式标记
2. **第二轮：目标语言重写**
   - 在不改变原意的前提下，用自然中文重写
   - 优先重构结构，不优先保留原文标题映射
   - 优先消除直译腔，不补充原文没有的信息
   - 继续保留代码、专名、日期、数字和确实承载信息的 Markdown 结构

第二轮不是自由改写，而是受约束的中文重写。

#### 2.1 技术文章写作结构（推荐）

```
1. 背景与动机     — 这个技术/问题是怎么出现的？解决什么痛点？
2. 核心原理       — 关键概念与机制，配合示意图或代码片段
3. 工程实践       — 如何在实际项目中落地，注意事项与坑
4. 延展思考       — 与其他技术的对比 / 适用边界 / 未来趋势
5. 参考资源       — 原文链接 + 延伸阅读（可选）
```

> ⚠️ 不是所有文章都需要五节。根据原文内容灵活取舍，保持结构清晰即可。

#### 2.2 frontmatter 与文件操作

1. 读取 `src/constants.ts`，以 `FRONTMATTER_TAGS` 作为 tags 的唯一来源
2. 从主题推导简洁的 kebab-case `slug`（英文优先）
3. 使用今天的日期作为 `date` 和 `lastUpdated`
4. 检查 `content/blog/YYYY-MM-DD--{slug}` 是否已存在：
   - 不存在 → 新建
   - 存在且为空 → 复用
   - 存在且非空 → 追加后缀（如 `-2`）避免冲突
5. 创建文件：`content/blog/YYYY-MM-DD--{slug}/index.mdx`
6. 正文使用**简体中文**
7. 正文开头（紧接 frontmatter 之后）插入引用说明块：

```mdx
:::note[引用说明]
本文基于 <作者名> 的文章整理与延展：[<标题>](<URL>)。
:::
```

#### 2.3 写作与翻译约束

写作时始终遵守下面几条：

1. 固定事实骨架：保留代码块语言、列表中的关键编号、引用关系和必要的 Markdown 结构，但不机械保留原文章节映射
2. 固定事实边界：不要补充原文没有明确支持的背景、评价和趋势判断
3. 固定术语一致性：正文、标题、小节标题和引用说明中的术语保持一致
4. 固定保留项：数字、日期、单位、人名、产品名、命令、代码、论文引用默认保留
5. **站内链接规范**：引用本站其他博客文章时，直接使用 slug（如 `/harness-engineering-agent-first`），不要添加 `/blog/` 前缀（错误示例：`/blog/harness-engineering-agent-first`）

如果需要重构，优先重构结构，不重构事实。

#### 2.3.1 小节标题约束

正文中的 `##` / `###` 小节标题默认使用**简洁、直接、结论式**写法，优先概括本节内容，不写作者动作或写作姿态。

规则：

1. 优先使用名词短语、判断句或并列短语，例如：
   - `实践建议`
   - `模式可以组合`
   - `先选对问题类型`
   - `给步骤加门禁`
2. 避免口语化、自述式、提问式或铺垫式标题，例如：
   - `如果让我压成一条实践建议`
   - `真正有用的，不是……而是……`
   - `为什么这件事值得关注`
   - `我最后的看法`
3. 小节标题默认不超过 `12` 个汉字；若必须保留英文术语，仍应尽量缩短
4. 标题优先服务读者扫描和定位，不承担文风表达任务
5. 如果正文已经表达转折、态度或限定条件，标题不要重复承担这些信息

#### 2.4 批判性复核与定向修订

第二轮重写完成后，不要立刻进入压缩或润色，先做一次**批判性复核**。这一步只产出问题清单，不直接重写全文。

至少检查下面六类问题：

1. **准确性与遗漏**
   - 有没有偏离原意、漏掉限制条件、把评论区观点写成作者原意
2. **术语一致性**
   - 标题、摘要、正文、引用说明中的关键术语是否统一
3. **直译腔 / 欧化中文**
   - 是否存在连接词过密、被动句堆积、名词堆叠、`是……的` 结构、抽象名词化、代词滥用
4. **比喻与语气**
   - 原文比喻、习语、情绪色彩和隐含判断是否被直译或压扁
5. **结构问题**
   - 是否还在按原文目录逐章复述，导致正文像“整理稿”而不是“作者读后判断”
6. **博客适配**
   - `description`、开头段和小节标题是否已经符合本站最近文章的判断密度与节奏

完成复核后，只做**定向修订**：

- 只改被复核明确指出的问题
- 优先局部修订，不整篇重写
- 如果某段只是“不够漂亮”但没有事实或结构问题，不要反复改写

#### 2.4.1 关键段落回译校对

回译是批判性复核的一部分，不要对整篇文章回译。默认只对关键段落做抽样校对：

- 标题和摘要
- 原文核心论点所在段落
- 定义、结论、限制条件
- 术语密集或容易误译的段落

校对目标只有两个：

1. 是否偏离原意
2. 是否遗漏关键限定条件

发现问题后优先局部修订，不整篇重写。

### Step 2.5 — 压缩与风格收口

在进入 `humanizer-zh` 之前，先做一次人工压缩，目标不是“再写漂亮一点”，而是去掉整理稿味道。

优先处理下面几类问题：

- 重复铺垫：同一个判断说两遍，保留一次即可
- 讲义式转场：例如“更重要的是”“换句话说”“这意味着”连续出现时，合并或删除
- 机械列表：不是所有观点都要 `1 2 3 4` 展开，能并句就并句
- 说明书摘要：`description` 不要写成“基于某文整理了什么”，要直接写文章最核心的判断

如果原文是中文评论、观点文或行业观察，不要硬套“背景 / 原理 / 实践”模板。更自然的结构通常是：

1. 先给出你最认同的一条判断
2. 再展开原文最关键的两到四个支点
3. 最后补自己的结论或取舍

如果来源是 Reddit、论坛、评论串这类“讨论型内容”，优先避免写成“原帖讲了什么 + 评论区讲了什么”的双栏式摘要，更推荐压成下面这种形态：

1. 先给出你最终留下的一条总判断
2. 再展开两到四个最关键的争议点或支点
3. 最后交代你的取舍、边界或保留意见

压缩目标：

- 优先减少“我在复述一篇文章”的感觉
- 改成“我读完后留下哪些判断”
- 在不丢事实的前提下，尽量缩短篇幅和段落长度
- 对讨论型来源，优先减少“楼主说 / 评论说”的并排转述，改成围绕争议点组织内容
- 工作过程里允许出现术语标记、可疑点标记或译者注，但这些默认不进入最终博客正文

#### 2.5.1 存量文章压缩模式

如果用户不是要求“从 URL 发布新文章”，而是要求：

- 压缩已有文章
- 去掉“逐章总结”“复述原文”的感觉
- 把文章收成更像作者自己的判断

则允许直接编辑现有 `content/blog/**/index.mdx`，不必走新建目录和新建文章流程。

此时优先遵守下面几条：

1. 如果正文接近“原文每个章节一个小节 + 每节一段总结”，优先重组为“一个总判断 + 两到四个关键支点 + 结论”，不要机械保留原始章节映射
2. 压缩时优先删除重复判断、讲义式转场和机械编号，不优先删核心论点
3. 可以合并段落、重写转场、调整小节结构，但不要补充原文没有明确支持的新判断
4. 如果用户没有指定篇幅，中文评论、观点文、行业观察类文章默认控制在约 `70-100` 行；以“信息密度高于结构对称”为优先目标
5. 压缩后的正文应更像“作者读完留下的判断”，而不是“对原文目录的逐项摘要”

### Step 2.6 — 内容润色（humanizer-zh）

在校验之前，调用 [humanizer-zh](./agents/skills/humanizer-zh) Skill 对正文（不含 frontmatter）进行润色，减少 AI 写作痕迹。

润色要求：

- 保持所有事实、链接、人名、日期、数字、代码片段和技术术语不变（除非明显有误）
- 保留「为什么 → 是什么 → 怎么用」的学习者结构，不删除引用说明块
- 优先使用自然、简洁的中文，避免模板化表达
- `humanizer-zh` 只负责语言收口，不负责新增事实、补背景或替代前面的质检步骤
- 参考博客里 1-2 篇最近发布、题材相近的文章，优先对齐句长、段落节奏和摘要写法
- 润色后进行 diff 检查，确保未引入新的不可核实的说法
- 如果 `humanizer-zh` 不可用，保留原文继续执行，不阻塞发布

对于评论帖、观点文、行业观察类文章，如果第一轮润色后仍然有明显“整理稿”气味，允许再做一轮**高压缩收口**：

- 继续删除重复判断和解释性复述
- 合并过于整齐、机械对称的段落
- 宁可少一点结构对称，也要保留判断密度
- 第二轮压缩后再次检查 diff，确认没有新增原文未支持的判断

### Step 3 — 校验

执行 blog-conventions.md 中的校验清单，并运行构建：

```bash
cd $BLOG_ROOT
pnpm build:astro
```

**校验门控：**

- [ ] 所有必填 frontmatter 字段均已填写
- [ ] tags 值与 `src/constants.ts` 完全一致（区分大小写）
- [ ] 目录名格式为 `YYYY-MM-DD--{slug}`
- [ ] 文件名为 `index.mdx`
- [ ] 无 schema 之外的 frontmatter 字段

发现问题先修复，再继续。

除构建外，再做一次发布前人工检查：

- [ ] 文章是否先讲"为什么重要"，再讲"是什么"和"怎么做"
- [ ] 是否还有明显的直译腔或句式贴源文的问题
- [ ] 最小术语表中的关键术语是否前后一致
- [ ] 是否已经完成批判性复核，并只修订了有证据的问题
- [ ] 关键段落是否已做回译校对或等价核对
- [ ] 是否误加了原文没有的背景判断
- [ ] 正文是否避免了按原文目录逐章复述
- [ ] `##` / `###` 标题是否简洁、直接，避免"如果让我……""真正……不是……而是……"这类自述式或讲义式标题
- [ ] 站内链接是否使用正确格式（直接用 slug，不含 `/blog/` 前缀）

---

### Step 4 — 提交并推送

```bash
cd $BLOG_ROOT
git status --short
git add content/blog/YYYY-MM-DD--{slug}/
git commit -m 'docs: add post "<文章标题>"'
git push origin main
```

commit 格式遵循 blog-conventions.md 的规定。

**提交边界：** 除非用户明确要求，只暂存新建的文章目录，不包含其他文件。

## 技术文章写作原则

| 原则           | 说明                                           |
| -------------- | ---------------------------------------------- |
| 学习者优先     | 先讲动机与背景，再进入细节，降低认知门槛       |
| 不翻译，要重构 | 用自己的语言重新组织，不做逐字翻译             |
| 代码要可运行   | 保留原文代码时注意版本与依赖说明               |
| 观点要有增量   | 「延展思考」部分需有真正新的洞见，不是简单复述 |
| 来源透明       | 始终保留引用说明，标注原作者与原文链接         |
